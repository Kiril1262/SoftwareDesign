# Принципи програмування, продемонстровані в цій лабораторній роботі

Цей проєкт демонструє кілька важливих принципів програмування. Нижче наведено список цих принципів з посиланнями на відповідні файли та рядки коду.

1. DRY (Don't Repeat Yourself / Не повторюйся)

Опис: Принцип DRY спрямований на зменшення дублювання коду шляхом повторного використання функцій, класів або модулів.
Приклад:

Функція session_start() використовується у кількох файлах PHP (наприклад, animal.php, report.php) для збереження даних сесії.

Дані про тварин, їжу та працівників зберігаються в сесії, а не дублюються в різних місцях.

2. KISS (Keep It Simple, Stupid / Зроби просто)

Опис: Принцип KISS закликає до простоти у проєктуванні коду для покращення його підтримки та читабельності.
Приклад:

У PHP-шаблонах для відображення тварин, їжі та працівників використовуються прості цикли foreach (animal.php).

Дані організовані у вигляді асоціативних масивів, що робить код гнучким і легким для розуміння.

3. SOLID принципи

a) Принцип єдиної відповідальності (SRP)

Опис: Кожен модуль або клас повинен мати одну відповідальність.
Приклад:

animal.php відповідає лише за відображення інформації про тварин.

report.php генерує звіт без зміни даних сесії.

b) Принцип відкритості/закритості (OCP)

Опис: Код повинен бути відкритим для розширення, але закритим для модифікації.
Приклад:

Додавання нових тварин, їжі чи працівників не потребує змін основної логіки, оскільки використовується сесійний масив (animal.php).

c) Принцип підстановки Лісков (LSP)

Опис: Підкласи повинні бути замінними для своїх батьківських класів.
Приклад:

Якщо використовувати ООП, клас Animal можна було б замінити підкласами Mammal або Bird без порушення логіки.

d) Принцип розділення інтерфейсів (ISP)

Опис: Клієнти не повинні залежати від інтерфейсів, які вони не використовують.
Приклад:

Використання асоціативних масивів у PHP дозволяє уникати примусу об'єктів містити непотрібні властивості.

e) Принцип інверсії залежностей (DIP)

Опис: Високорівневі модулі не повинні залежати від низькорівневих модулів.
Приклад:

Використання сесійних даних для зберігання інформації дозволяє уникати прив’язки до конкретної бази даних.

4. YAGNI (You Ain't Gonna Need It / Вам це не знадобиться)

Опис: Не слід додавати функціональність, яка наразі не потрібна.
Приклад:

Проєкт не використовує складну систему керування базою даних, оскільки для поточних завдань достатньо сесійного збереження (animal.php).

5. Композиція замість успадкування (Composition Over Inheritance)

Опис: Краще використовувати композицію, ніж глибоку ієрархію класів.
Приклад:

Замість успадкування використовується масив для зберігання даних про тварин, що робить код гнучкішим (animal.php).

6. Програмування на рівні інтерфейсів, а не реалізацій (Program to Interfaces, Not Implementations)

Опис: Код повинен залежати від абстракцій, а не від конкретних реалізацій.
Приклад:

Якщо б використовували класи, варто було б зберігати тварин через інтерфейс Animal, а не конкретні класи Lion чи Tiger.

7. Швидке виявлення помилок (Fail Fast)

Опис: Система повинна швидко припиняти роботу, якщо виявлено некоректні дані.
Приклад:

У report.php скрипт завершується (die()), якщо відсутні дані сесії, запобігаючи непередбачуваній поведінці (report.php).
